# Security

Security is not a feature — it is a foundation. Every design decision starts here.

---

## Threat Model Summary

| Threat                              | Mitigation                                              |
|-------------------------------------|---------------------------------------------------------|
| Unauthenticated score manipulation  | Server-side ownership validation on every write         |
| Share token forgery                 | HMAC-signed tokens; server validates signature         |
| JWT tampering                       | RS256 / ES256 signature verification via Authentik JWKS |
| CSRF                                | SameSite=Strict cookies; no cookie-based session auth  |
| XSS                                 | Angular template escaping; strict CSP header           |
| SQL injection                       | EF Core parameterised queries only; no raw SQL          |
| Insecure direct object reference    | All resource access checks user/token ownership        |
| Enumeration attacks                 | Resources use UUID v4 IDs; no sequential IDs exposed   |
| Brute-force share tokens            | Rate limiting per IP + per endpoint                    |
| Sensitive data exposure             | No PII in logs; tokens never in URLs                   |

## Authentication — Authentik OIDC

### Flow

1. Frontend initiates OIDC Authorization Code + PKCE flow via `angular-oauth2-oidc`.
2. Authentik issues ID token + access token.
3. Frontend stores tokens in memory (not `localStorage`) via the OIDC library's session management.
4. Access token is attached to every API request via the `AuthInterceptor`.
5. Backend validates JWTs by fetching Authentik's JWKS endpoint (`/.well-known/jwks.json`).

### Token Storage Rules

| Token        | Storage          | Reason                                          |
|--------------|------------------|-------------------------------------------------|
| Access token | In-memory        | Short-lived; not persisted across tabs           |
| Refresh token| `sessionStorage` | Tab-scoped; cleared on close                    |
| Session token (anonymous) | `localStorage` | Intentional; keyed by counter ID; not a credential |

**Never store JWTs in `localStorage`** — they are accessible to any script in the page.

### Session Token (Anonymous Users)

- Generated by the server on counter creation.
- UUID v4 — 122 bits of entropy.
- Stored server-side hashed (SHA-256); stored client-side as plaintext in `localStorage`.
- Sent via `X-Session-Token` header — never in the URL.
- Invalidated on counter claim by authenticated user.

## Authorisation — Server-Side Enforcement

### Principle: Deny by Default

Every API endpoint must explicitly grant access. The default policy is `[Authorize]`.
Endpoints that allow anonymous access are explicitly decorated with `[AllowAnonymous]`
and undergo additional review to confirm no sensitive data is exposed.

### Ownership Checks

For every counter or tournament write operation:

```csharp
// Example — never skip this pattern
var counter = await _repo.GetByIdAsync(id);
if (!_authorizationService.CanEdit(currentUser, sessionToken, counter))
    return Results.Forbid();
```

`CanEdit` checks:
1. If the user is authenticated — does the counter belong to their account?
2. If anonymous — does the `X-Session-Token` header match the stored hash for this counter?
3. If a share token — does it have `edit` scope and is it still valid?

See `docs/ROLES_PERMISSIONS.md`.

## Share Token Design

```
shareToken = base64url(counter_id + ":" + scope + ":" + expiry_unix_ts)
           + "." + HMAC_SHA256(server_secret, payload)
```

- `server_secret` is an environment variable — never in source code.
- Tokens are also stored in the database for revocation lookups.
- The server checks both signature validity AND database presence (revocation).
- Scope values: `read`, `edit`.

---

## Transport Security

- All traffic is HTTPS — enforced at the Nginx reverse proxy level.
- HSTS header with `max-age=31536000; includeSubDomains`.
- Nginx TLS config: TLS 1.2 minimum, TLS 1.3 preferred; strong cipher suites only.

---

## HTTP Security Headers (Nginx)

```nginx
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
add_header X-Content-Type-Options "nosniff" always;
add_header X-Frame-Options "DENY" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Content-Security-Policy "
  default-src 'self';
  script-src 'self';
  style-src 'self' 'unsafe-inline' https://fonts.googleapis.com;
  font-src 'self' https://fonts.gstatic.com;
  connect-src 'self' wss:;
  img-src 'self' data:;
" always;
```

Adjust `connect-src` to include the Authentik domain for OIDC discovery.

## Input Validation

| Layer     | Mechanism                          | What it covers                         |
|-----------|------------------------------------|----------------------------------------|
| Frontend  | Angular Reactive Forms + validators| UX feedback; not a security boundary   |
| Backend   | FluentValidation on every command  | Length, format, enum values, ranges    |
| Database  | NOT NULL, CHECK constraints        | Last-line-of-defence                   |

**The frontend validator is for UX only. The backend must never trust frontend validation.**

## Rate Limiting

Implemented via ASP.NET Core Rate Limiting middleware:

| Endpoint                        | Limit                      |
|---------------------------------|----------------------------|
| `POST /api/counters`            | 10 per minute per IP       |
| `POST /api/counters/*/share`    | 5 per minute per user/IP   |
| `POST /api/counters/join/*`     | 20 per minute per IP       |
| All other write endpoints       | 60 per minute per user     |
| All read endpoints              | 120 per minute per IP      |

## Logging and Monitoring

- Use structured logging (Serilog with JSON output).
- Log: authentication events, authorisation failures, share token creation/revocation.
- Never log: JWT contents, session tokens (even hashed), user passwords, PII.
- Log entries include: `timestamp`, `level`, `requestId`, `userId` (if authenticated), `endpoint`, `statusCode`.

## Dependency Security

- `dotnet list package --vulnerable` runs in CI on every PR.
- `npm audit` runs in CI on every PR.
- Dependabot or Renovate configured for automated patch PRs.

## Security Review Checklist (run before every PR merge)

- [ ] No secrets in source code or `.env` files committed.
- [ ] New endpoints have explicit auth policy defined.
- [ ] Any `[AllowAnonymous]` has a written justification in the PR.
- [ ] User-supplied data is validated in FluentValidation before use.
- [ ] No raw SQL added.
- [ ] No new `localStorage` usage for tokens.
- [ ] Share tokens, if any, go through `ShareTokenService` — not ad-hoc generation.
- [ ] Logging added, and it does not log sensitive fields.
